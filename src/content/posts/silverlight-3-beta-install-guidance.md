<p>If you are like any other developer, including me, you probably <a href="http://silverlight.net/getstarted/silverlight3/default.aspx">disregard most warnings</a> and are usually the same type that keeps clicking next when installing things without paying attention to detail.  That’s okay I do it too.</p>
<p>In the release of <strong>Silverlight 3</strong> Beta, we noted that this is a developer release and that no “Go Live” licensing was going to be available for this release.  What this means is that we don’t recommend putting things in production as we’ve not exposed or wired up the <em>end-user runtime</em> for <a href="http://silverlight.net/">Silverlight</a> 3.  The only way people can view Silverlight 3 content during this Beta phase is by having one of the versions of the developer runtime for either Windows or Mac installed.  We’ve not integrated the end-user upgrade/install into the template for now.  So what does this mean to you when you put a sample out for people to see in your organization or perhaps if you have a preview of a product or something and users visit this application.  Here’s a few scenarios for you – keeping in mind the information below is related to the application being build in Silverlight 3: <a href="#noversion">No Silverlight installed</a>, <a href="#sl3installed">Silverlight 3 installed</a>, <a href="#prevversion">Silverlight 1/2 installed</a> and a <a href="#mime">note about the plugin MIME type</a>.</p>
<h2><a href="noversion"></a>Scenario: No Silverlight Installed</h2>
<p>If the user has <em>no previous version of Silverlight</em> installed at all, they’ll be presented with whatever your default “not installed” experience is.  If you’ve don nothing to customize this, shame on you :-).  The default Visual Studio templates will simply provide you with a static image.  That link in the templates links to a “coming soon” page which describes the situation.  You can view that page here: <a href="http://go.microsoft.com/fwlink/?LinkID=141205">Silverlight 3 coming soon page</a>.  If the user installs one of the developer runtimes then they’d be able to review your application.</p>
<blockquote>
<p><strong>NOTE:</strong> People with Silverlight 3 Developer runtimes cannot “upgrade” to an end-user runtime.  This means that if an end-user (non-developer) installs the developer runtime, then when Silvelright 3 releases they will have to manually uninstall the developer runtime to get the update of the released runtime.  If you’re a developer, you probably don’t care and future developer runtimes can install on top of other developer runtimes.  This is one of the driving reasons for no Go Live support at this time.</p>
</blockquote>
<p>You should inform your viewers of your beta sample of the scenario and what they may have to do in the future.</p>

<h2><a href="prevversion"></a>Scenario: Silverlight installed (any released version)</h2>
<p>Ah, this is the tricky one.  If the user has any released version of Silverlight (1.0 or 2), the experience they see at your beta sample will be driven off of two properties as you instantiate the plugin: <em>minRuntimeVersion</em> and <em>autoUpgrade</em>.  Let’s look at what will happen and assume that the user has Silverlight 2 installed.</p>
<p><em>minRuntimeVersion=”3.0.40307.0” and autoUpgrade=”true”</em> – if you do nothing else, the user will see this:</p>
<p><img title="Silverlight default auto upgrade" alt="Silverlight default auto upgrade" src="http://storage.timheuer.com/slautoupgradetrue.png" /></p>
<p>This is bad on a few levels.  First because it isn’t a customized experience, it is the default.  Second because the “install” link will take them to the <em>latest released runtime</em> – which they already have most likely.  What happens?  The user feels like they are in and endless loop of Visit Site –&gt; prompted to upgrade –&gt; seemingly install –&gt; visit site –&gt; prompted to upgrade –&gt; repeat.  This will frustrate your beta visitor and confuse you :-).  </p>
<p>You should always customize the user experience for the end user.  For beta, here’s what I recommend doing to ensure they aren’t confused with the goals being the following:</p>
<ul>
    <li>Detect if the user needs to upgrade to see your beta sample </li>
    <li>Inform them of the situation and that your sample is beta </li>
    <li>Direct them to the coming soon information page for Silverlight 3 </li>
</ul>

<p>The error code of 8001 means that an upgrade is required.  Since we know we have a beta app and we know that the user has some version of Silverlight installed, but not the beta version.  Replace the container inner content with some HTML explaining that your application is a sample application and direct them to the coming soon page for more information.  This way your users aren’t sent in an endless loop, get confused, and hate you.</p>
<h2><a href="mime"></a>Why isn’t the &lt;object&gt; type attribute “application/x-silverlight-3”?</h2>
<p>A common misconception is that the <em>data</em> and <em>type</em> attributes on the plugin object directly correlate to the runtime version of the plugin you are using.  This is not true.  Just because you have a Silverlight 3 application does not mean you need to change it to <em>type=”application/x-silverlight-3”</em> and in fact you’ll have problems if you do!  These attributes refer to the plugin MIME type and not the runtime version.  You would still use <em>application/x-silverlight-2</em> for a Silverlight application (in fact you could use <em>x-silverlight</em> if you wanted to handle edge cases of Silverlight 1.0).</p>
<h2>Summary</h2>
<p>Beta software is fun and confusing.  It’s fun for us developers because we’re technically astute, understand what beta means, and are probably accustomed to working with others’ beta software all the time.  For end users, it can be confusing because they see things/links/articles and want to check it out, but may not have the required software.  As developers we can do things to help our beta visitors understand the scenario much clearer and do our job to inform.</p>
<p>So if you are showing Silverlight 3 beta samples, please take an extra moment to follow this guidance and trap the upgrade error code and handle it with information to inform the user.</p>
