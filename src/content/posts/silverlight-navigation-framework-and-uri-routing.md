<p>One of the new features in <a href="http://silverlight.net/"><strong>Silverlight</strong></a><strong> 3</strong> is providing an <strong>application navigation framework </strong>via the Frame and Page controls in the APIs.  If you saw <strong><a href="http://www.timheuer.com/blog/archive/2009/03/18/silverlight-3-whats-new-a-guide.aspx">my guide to Silverlight 3</a></strong>, you may have seen the <a href="http://www.timheuer.com/blog/archive/2009/03/18/silverlight-3-whats-new-a-guide.aspx#nav">section on navigation</a> which describes the functionality and as well has a link to a video tutorial about it.</p>

<p>A request to <em>About-Us</em>, <em>History</em> or <em>About</em> would navigate to <em>/Views/AboutPage.xaml</em> regardless.  This provides you some flexibility and granularity in your implementation and also provides additional SEO points for your content.</p>
<p>I really like the UriMapper in the navigation framework and have been lobbying to change the default templates Visual Studio spits out to use the UriMapper by default.  It provides you with a cleaner (and perhaps shorter) URI as well as obfuscates your “code” from your URI logic.  Be sure to check out the video if you aren’t familiar with the framework at all.</p>
<p>I hope this helps and would love to know what you think about the navigation framework?</p>
