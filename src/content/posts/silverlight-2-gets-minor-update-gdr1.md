<p>Yesterday a minor update to the <a href="http://silverlight.net">Silverlight</a> 2 runtime was released.  You may see terms of “GDR 1” floating around.  That’s pretty much an internal term at Microsoft referred to as “general distribution release” – yeah, I know we have a lot of random terms.  Think of it just as an incremental update.  If you are curious for Silverlight’s entire release history, you can <a href="http://tinyurl.com/slrelhist">view that here</a>.</p>  <p>So what’s in it?  Well, nothing earth shattering if you were expecting <em>feature</em> updates.  There were some needed fixes based on customer feedback that we included in this update (which is version 2.0.40115.0).  Basically I would bucket it into a simple list for you:</p>  <ul>   <li>Various accessibility fixes relating to UIAutomation </li>    <li>Fixes for certain anti-virus vendors scanning algorithms </li>    <li>Much needed fix for OSX platform when users modify their font locations (i.e., people with font management tools usually) </li>    <li>Bug in IsolatedStorage quota increasing when the user’s display language is set to one that Silverlight has not been localized to </li> </ul>  <p>That’s about it…no new controls, nothing suddenly going to make you amazed and shocked.  But it was a needed release to fix these issues.  </p>  <p>So what should you do?  Depends.  </p>  <p><em><strong>If you are a consumer…</strong></em></p>  <p>You really shouldn’t need to do anything unless you are specifically experiencing one of the issues above.  Web developers using Silverlight and needing their user’s to have this update will update their site and let you know.  There isn’t a huge need to rush out as an <em>end-user</em> and download the update.  Besides, if you have it configured to automatically update, then in due time you’ll get the update through that mechanism and won’t have to do anything.  If you absolutely want to be on top of things, then feel free to grab the latest updated runtime by <a href="http://go.microsoft.com/fwlink/?LinkId=124807">visiting the download link</a>.</p>  <p><em><strong>If you are a developer…</strong></em></p>  <p>If you are directly affected by the changes above then you should update your application.  There are really two things you need to do:</p>  <ul>   <li><a href="http://go.microsoft.com/fwlink/?LinkID=119972">Install the updated developer runtime</a> for your development environment.  There is no new Visual Studio tools update for this release, just install the updated developer runtime to be sure you are targeting the update in your development/debugging.  *New* developers installing the <a href="http://go.microsoft.com/fwlink/?LinkId=129043">Silverlight Tools for Visual Studio</a> will automatically get the updated developer runtime during that install.  If you are behind a proxy you can follow the <a href="http://timheuer.com/blog/archive/2008/09/29/install-silverlight-2-rc0-offline.aspx">“offline” installation method</a>. </li>    <li>Update your applications: re-compile/distribute your code and update your object tag versioning. </li>    <li>For Mac dev/<a href="http://blogs.msdn.com/peterlau/archive/2008/10/14/remote-debugging-a-silverlight-application-on-mac-osx.aspx">debug</a>: if you need the updated <a href="http://go.microsoft.com/fwlink/?LinkID=119973">Mac developer runtime you can get that here</a>. </li> </ul>  <p>For the latter note this is a simple change.  Update your “MinRuntimeVersion” attribute in your hosting page/content to ensure that end users have the latest runtime version for your application.  Again, <em>you should only really need/want to do this proactively if you (or your users) are directly affected by the updates in this release!</em>  Here’s how you’d do it.</p>  <blockquote>   <p>NOTE: Even though you update the developer runtime the Visual Studio project templates are not updated to set the default runtime version to the updated version so each new Silverlight project using those templates will still reference minRuntimeVersion=”2.0.31005.0” – if you want to change that you can modify the template.</p> </blockquote>  <p><strong><u>UPDATE: How to modify the default value of the template</u></strong></p>  <p>I’ve received some errors myself after changing this, so while it sounded like a good idea :-), I can’t recommend altering this reg value to the updated version – if I find out why, I’ll post here.</p>  <p><strike>I got some questions about exactly how you would get the default web templates to change the generated test pages to emit the updated version number.  Since these are pages in a <em>web</em> project and not the
