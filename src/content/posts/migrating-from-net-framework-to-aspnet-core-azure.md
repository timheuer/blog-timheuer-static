<p>Reflecting back on this blog I realized it’s been 16 years of life here.&#160; It hasn’t always been consistent content focus on tech over the early years versus a more random outlet of my thoughts (and apparent lack of concern over punctuation and capitalization).&#160; Some months had more volume and as my career (and perhaps passions) changed some had lower volume.</p>  <p><img title="Month comparison of blog post quantity" style="border: 0px currentcolor; border-image: none; margin-right: auto; margin-left: auto; float: none; display: block; background-image: none;" border="0" alt="Month comparison of blog post quantity" src="https://storage2.timheuer.com/monthcompare.png" width="950" height="358" /></p>  <p>I’ve enjoyed getting back in to posting more recently and finding more time (and again, perhaps the passion) to do so.&#160; I think also with newer various outlets of social media, my personal passions are posted elsewhere now like Instagram (if you want to follow my escapades on the bike mostly).&#160; Recently this summer in 2019 I switched job roles at Microsoft back into program management with the .NET team.&#160; I’m focusing on a few different things but having spent so much time in UI frameworks on the client side for so long, I missed some waves of changes in ASP.NET and needed to re-learn.&#160; I spent the first month of my new role doing this and exploring the end-to-end experiences.&#160; Instead of building a To-do app, I wanted to have some real scenario for me to work with so I set off to migrate my blog…it was time anyway as just that week I had received warnings on my server about some errors.&#160; I could avoid this no longer.&#160; It wasn’t an easy path, but here was my journey.</p>  <h2>Existing website frameworks</h2>  <p>My blog started in 20-Aug-2003 and was built using Community Server (from Telligent) initially.&#160; That quickly forked into a product called .TEXT from Scott Watermasysk and I moved to using that as it was solely content management and not forums or other things I didn’t need.&#160; I stayed on .TEXT for as long as it lived until again another fork happened.&#160; This time the .NET ecosystem around Open Source was improving and this fork was one of those projects.&#160; Phil Haack, along with others, created <a href="https://github.com/haacked/subtext">SubText</a> which was initially a pretty direct fork, but quickly evolved.&#160; I wasn’t much interested in the code at this point, but wanted to follow ‘current’ frameworks so I moved to SubText.&#160; All along this path it was easy because these migrations were similar, using the same frameworks and similar (if not same) data structures.&#160; The SubText site was an ASP.NET 2.0 site using SQL Server as the data store.&#160; Over time this moved to .NET 3.5 but not much more after that (for me at least).&#160; </p>  <p>Over time I made a few adjustments to the SubText environment for me, never really concerning myself about the source, but just patching crap in random binaries that I’d inject into the web.config.&#160; My last one was in <a href="https://timheuer.com/blog/adding-twitter-summary-cards-to-blog-content/">2013 to support Twitter cards</a> and it was a painful reminder at the time that this site was fragile.&#160;&#160; By this time as well the SubText project itself was fragile and not really being maintained as ASP.NET had moved to newer things like MVC and such.&#160; The writing was on the wall for me but I ignored it.</p>  <h2>Hosting environment</h2>  <p>In addition to the platform/framework used, I was using an interesting hosting setup.&#160; Well, not abnormal really considering at the time in 2003 there was no ‘cloud’ as we know it today.&#160; I had a dedicated box (1U server) hosted at my own data center (I was managing, among other things, a data center rack at the time).&#160; This was running Windows Server 2000 and whatever goop that came with.&#160; Additionally this was SQL Server Express [insert some old version here].&#160; I had moved on to another job and after a period of time I needed to move that server.&#160; I was using the server for more than just my site, running about 10 other WordPress blogs for my community, my wife’s business, and various other things.&#160; WordPress sites were constantly being attacked/hacked due to vulnerabilities in WordPress and leading to my server being filled with massive video porn files and me not knowing until my site was down then I had to login remotely and clean crap up.&#160; Loads of fun.&#160; I eventually moved that server to a co-located environment at GoDaddy still maintaining a dedicated 1U server for me.&#160; It was nice having direct access to the server to do whatever I wanted, but I was quickly not needing that type of hands-on configuration anymore…but still dealing with the management.</p>  <p>During each of these moves I was just moving folders around.&#160; I had no builds, no original source code reliably, etc.&#160; “Fixing” things was me writing new code and finding interesting ways to redirect some SubText functionality as I didn’t have the source nor was interested in digging up tooling to get the source to work.&#160; I never upgraded from Windows 2000 server and was well beyond support for things I was doing.&#160; When I wanted to upgrade the OS at GoDaddy, I was faced with “Sure we’ll set up a new server for you and you migrate your apps” approach.&#160; So again I was going to have to re-configure everything.&#160; Another nightmare waiting and I just put it off.</p>  <h2>To the Cloud!</h2>  <p>My first step was moving data to the cloud.&#160; I <a href="https://timheuer.com/blog/migrating-existing-sql-server-to-sql-azure/">wrote about this when I did this task back in 2012</a>.&#160; I quickly learned that if my site wasn’t on Azure as well and with the traffic I was still getting, the egress costs were not going to be attractive for me.&#160; A few years ago I went about moving just my blog app to Azure App Service as well.&#160; Not having anything to build, this was going to be a fun ‘deployment’ where I needed to copy a lot of things manually to my App Service environment.&#160; I felt dirty just FTP-ing in to the environment and continually trying things until it worked.&#160; But it eventually did.&#160; I had my .NET Framework SubText app running on App Service and using Azure SQL.&#160; The cool thing about Azure SQL is the <a href="https://timheuer.com/blog/using-machine-learning-to-tune-your-sql-database-in-azure/">monitoring and diagnostics</a> it provides.&#160; I immediately was met with a few recommendations and configuration changes I should make and/or it automatically made on my behalf.&#160; That was awesome.&#160; I did have one stored procedure from SubText that was causing all kinds of performance havoc and contributing to me hitting capacity with my chosen SQL plan requiring me to bump up to the next plan and more costs.&#160; Neither of which I wanted to do.&#160; And due to what I mentioned prior about not having SubText buildable I couldn’t reliably make a change to the stored proc without really changing the code that called it.&#160; Just another dent in the plan.&#160; I needed a real migration plan.</p>  <h2>Migrate to ASP.NET Core</h2>  <p>I mentioned that in my new role I needed to spend time re-learning ASP.NET and this was a perfect opportunity.&#160; I decided to dedicate the time and ‘migrate’ to ASP.NET Core.&#160; Why the air quotes?&#160; Because realistically I couldn’t migrate anything but data.&#160; I did not have reliably building source for SubText and despite that it was WebForms and I didn’t know what I might be getting myself in to.&#160; I needed a new plan, which meant a new framework and I went looking.&#160; Immediately I was met with recommendations that I should go static sites, that Jekyll and GitHub pages are the new hotness and why would I want anything else.&#160; I don’t know, for me, I still wanted some flexibility in the way I worked and I wasn’t seeing that I’d be able to get what I want out of a static site approach.&#160; I wanted to move to ASP.NET Core solutions and found a few frameworks that looked attractive.&#160; Most were in varying states and others felt just too verbose for my needs.&#160; I landed on a recommendation to look at <a href="https://github.com/madskristensen/Miniblog.Core/">miniblogcore</a>.&#160; This was the smallest, simplest, most understandable solution to my needs that I found.&#160; No frills, just render posts with some dynamicism.</p>  <p>I did not even attempt to migrate any of my existing ASP.NET WebForms code or styling as modern platforms were using Bootstrap and other things to do the site, so that was where I needed to start.&#160; I spent a good amount of time working on the simple styling structure for a few things and learning MVC in the process to componentize some of the areas.&#160; I added a few pieces of customization on the miniblog source, adding <a href="https://timheuer.com/search">search routing</a> (using Google site search), a <a href="https://timheuer.com/blog/timeline">timeline/calendar view</a> thanks to <a href="https://www.telerik.com/aspnet-core-ui">Telerik controls</a>, <a href="https://timheuer.com/blog/categories">category browsing</a> (although mine is horrendous due to waaay too many categories used over the years), using Disquss for commenting, adding SyntaxHighlighter for code formatting support, an image provider for my embedded images during authoring, and a few other random things.&#160; I wrote a little MVC controller to do the data migration once from SQL to the XML file-based storage that MiniBlog uses.&#160; That was a lot simpler than I thought it would take to migrate the data to the new structure.&#160; Luckily my old blog had ‘slug’ support and this new one had it as well, so the URI mapping worked fine, but now I had to ensure the old routing would work.&#160; I had to play around with some RegEx skills to accomplish this but in the end I found a pattern that would match and implemented that in my routing, using proper redirect response codes:</p>    <pre class="brush: csharp; toolbar: false; highlight: [6];">// This is for redirecting potential existing URLs from the old Miniblog URL format
// old subtext non-slugged &amp; slugged
// https://timheuer.com/blog/archive/2003/08/19/145.aspx
// https://timheuer.com/blog/archive/2015/04/21/join-windows-engineers-at-free-build-events-around-the-world-xaml.aspx
[Route(&quot;/post/{slug}&quot;)]
[Route(&quot;/blog/archive/{year:regex(\\d{{4}})}/{month:regex(\\d{{2}})}/{day:regex(\\d{{2}})}/{slug}.aspx&quot;)]
[HttpGet]
public IActionResult Redirects(string slug)
{
    // if the post was a non-named one we need to append some text to it otherwise it will think it is a page
    // if (slug doesn't contain letters) { redirect to $&quot;post-{slug}&quot; }
    var newSlug = slug;
    var isMatch = Regex.IsMatch(slug, &quot;^[0-9]*$&quot;);
    if (isMatch) newSlug = $&quot;post-{slug}&quot;;

    return LocalRedirectPermanent($&quot;/blog/{newSlug}&quot;);
}
</pre>



<p>That ended up being remarkably simpler than I thought it would be as well.&#160; This alone was causing me stress to maintain the URIs that had existed over time and using ASP.NET routing with RegEx I got what I needed quickly.</p>

<p>Moving to Azure App Service once this was all done was simple.&#160; When I first moved ASP.NET Core 3.0 wasn’t yet available so I had to deploy as a self-contained app.&#160; This isn’t difficult though and in some cases may be more explicitly what you want to do.&#160; I wrote <a href="https://timheuer.com/blog/deploy-aspnet-core-applications-using-self-contained-dotnet-core">how to Deploy .NET apps as self-contained</a> so you can follow the steps.&#160; This basically is a ‘bring the framework with you’ approach when the runtime might not be there.&#160; Azure App Service now has .NET Core 3.1 available though so I no longer have to do that, but good to know I can test future versions of .NET by using this mechanism.</p>

<h2>Summary</h2>

<p>So what did I learn?&#160; Well, not having source for your apps you care about hurts.&#160; I didn’t even get a <em>chance</em> to actually attempt to truly migrate SubText to ASP.NET Core because I had let my implementation rot for so long.&#160; I have become such a huge believer in DevOps now it’s unreal.&#160; I won’t do a simple project even without it.&#160; The confidence you gain when your projects have continuity through automation is amazing.&#160; My new blog app is fully run on DevOps and deploys using that as well…I just commit changes and they are deployed when I approve them.&#160; I learned that even though this was ‘just a blog’ it was a fairly involved app with separation of user controls and things.&#160; It didn’t need to be so complex, but it was and I’m glad for MiniBlog not being so complex.&#160; The performance of my content site and costs are much more manageable now and my stress is reduced knowing that should anything happen I’m in a better place for restoring a good state.&#160; My biggest TODO task I think is re-thinking the XML-based data store though.&#160; This actually is the one thing causing me some DevOps pain because the ‘data’ is content within the web app and when using slot-staging deployment that doesn’t work well.&#160; Azure has a way to <a href="https://docs.microsoft.com/en-us/azure/app-service/containers/how-to-serve-content-from-azure-storage">use Azure Storage as a mounted point to serve content from in your App Service</a> though and I’ve started to try that with some mixed results so far.&#160; Using this approach separates my app from my data and allows for more meaningful deployment flows and data backup.&#160; I’ve also explored using an Azure Storage provider for my data layer, but the method for how the initial cache is build in MiniBlog right now makes this not a great story due to startup latency when you have 2,000 posts to retrieve from blob container.&#160; I’m still playing around with ideas here, so if you have some I’d love to hear (dasBlog users would hit similar concerns).</p>

<p>I’m happy with where I landed and hope this keeps me on a path for a while.&#160; I’ve got a simple design, responsive design, easy-to-maintain source code, all the features I want (for now), no broken links (I think), works with my editing flow (Open Live Writer), and less stress worrying about a server.&#160; I’ve already updated to ASP.NET Core 3.1 and it was a simple config change to do that now that my setup is so streamlined.&#160; </p>
