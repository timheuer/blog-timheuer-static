<p>We were all jumping for joy when <a href="http://silverlight.net"><strong>Silverlight</strong></a><strong> </strong>2 beta 1 was released and the ability to connect to services was more readily/easily available to us.  For discoverable services that provided a <strong>WSDL </strong>we were quickly able to implement them using the <em>Add Service Reference</em> capability in <strong>Visual Studio 2008</strong>.  Beta 2 brings a few changes to the world of services that you should know about.  I’ll do my best to recap some of them here.</p>  <p><strong><u>Generating a WCF Service</u></strong></p>  <p>In beta 1 when we created a WCF service for use in Silverlight, we used the “WCF Service” template in Visual Studio (assuming you used Visual Studio).  This was fine and created a standard WCF service for us.  There were a few changes that we had to make to ensure that our Silverlight application could consume it in an acceptable manner.  First, we had to change the binding configuration in ASP.NET web.config from <em>wsHttpBinding</em> to <em><strong>basicHttpBinding</strong></em> as Silverlight only supports that binding type right now.  Secondly, we might have had to add capabilities to enable ASP.NET compatibility support depending on what we were doing with the service.  In beta 2, this process gets a bit simpler for services specifically built for Silverlight.  After you install the tools for Visual Studio, you now get a new item type:</p>  <p><img src="http://s3.amazonaws.com:80/timheuer-img/wcfservice-1.png" /></p>  <p></p>  <p></p>  <p></p>  <p></p>  <p></p>  <p></p>  <p></p>  <p></p>  <p>It is important to note that the <em>WCF Service</em> template is still a perfectly acceptable one to choose, you just have to ensure to make those changes accordingly.  The new <em><strong>Silverlight-enabled WCF Service</strong></em> basically does those for us as well as add the ASP.NET compatibility attributes for us in the code.  Additionally, the traditional interface/implementation is simplified into a single class.  Again, the other ways are still valid, but for services specifically built for Silverlight, this might be an easier route to get them done.</p>  <p><strong><u>Cross-domain policy file updates</u></strong></p>  <p>Cross-domain restrictions still apply within beta 2 and the same rules apply.  There is one subtle change that is required to your <em><strong>clientaccesspolicy.xml</strong></em> file that is required.  In the <em>allow-from</em> node of the policy file, the attribute <em><strong>http-request-headers</strong></em> is now required.  If your service is an open/public one then specifying “*” is probably acceptable (you’ll have to be the judge of that.  If you only wanted to allow specific headers (besides the blacklisted ones) you can provide those in a comma-separated list and can use wildcards as well.  For example you could use <em>X-MyApp-*</em> if you wanted.</p>  <p>Another thing to note about the support for Adobe’s <em>crossdomain.xml</em> policy file is one thing we found in interpretation of the policy template.  Previously Flash was a Macromedia product and as such that file is adorned with a DOCTYPE that represents a schema with macromedia in it.  Well, Adobe has changed the schema a little bit and also updated the DOCTYPE to reflect Adobe as the authority.  Right now, Silverlight still expects to validate the macromedia declaration.</p>  <p><strong><u>ServiceReferences.ClientConfig</u></strong></p>  <p>In beta 1, when you performed the <em>Add Service Reference</em> operation a file named <em>ServiceReferences.ClientConfig</em> was created and had some configuration information in it.  This file, however, wasn’t really used.  In beta 2, this configuration file can be shipped with your XAP and used as configuration.  It provides a subset of WCF configuration.  Refined details of those settings are in the SDK, but I thought it might be helpful to know.</p>  <p><strong><u>Change to WebClient</u></strong></p>  <p>In beta 1, <strong>WebClient</strong> was the easiest library to use in accessing non-discoverable services.  One challenge was that it only supported GET verb requests.  In beta 2 WebClient has changed to enable <strong>POST</strong> verb actions as well.  the <strong>UploadStringAsync</strong> function will send the request but the endpoint URI must be a URI that accepts the POST verb.</p>  <p>In addition, WebClient is now callable on a background thread in addition to the UI thread.  This may come in handy for some situations.</p>  <p>I see these as small but helpful changes.  Most are based on feedback we received from beta 1 customers and community, so thank you for that feedback.  I hope this helps!</p>  <div class="wlWriterSmartContent" id="scid:0767317B-992E-4b12-91E0-4F059A8CECA8:de8e057a-8457-4bc0-95f3-a848e9c2cbf8" style="padding-right: 0px; display: inline; padding-left:
