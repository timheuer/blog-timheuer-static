<p>If your a developer that spends any time with web services, you’ll basically <em>probably </em>end up working in one of two camps: <strong><a href="http://en.wikipedia.org/wiki/REST">REST-</a></strong> or <a href="http://en.wikipedia.org/wiki/SOAP"><strong>SOAP-based</strong></a><strong> </strong>services.  Now with SOAP services you’re likely used to having a WSDL document describing the service, providing type definitions, etc. – something you can use developer tools like Visual Studio to <em>Add Service Reference</em> and get a strongly-typed object model to work with.</p>  <p>If you’ve been doing <a href="http://silverlight.net/"><strong>Silverlight</strong></a> or ASP.NET (or any other technology really) development with mashup services, you probably have been working with REST-based services.  These are services that don’t self-describe themselves in a manner like SOAP with WSDL does.  Often I’ve found that really only the larger REST service providers provide good documentation for their services.  As a consumer of a REST service, you’re at the mercy of the documentation to understand the structure of the requests/responses that you’ll be working with…at times that can be frustrating.  If you are like me, you’ve probably either found someone else’s wrapper to the API or tried to work some other method to avoid spelunking the XML nodes.</p>  <p>If you don’t need to take on the full wrapper that you may have found someone already doing and maybe just need to consume something quick or whatever, enter <em><strong>Paste XML as Types</strong></em>.  Located in the <a href="http://aspnet.codeplex.com/Release/ProjectReleases.aspx?ReleaseId=24644"><strong>WCF REST Starter Kit Preview 2</strong></a>, this is a Visual Studio new option under the Edit menu.  Let’s take a look at an example.</p>  <p><a href="http://twitter.com/timheuer">Twitter</a> sounds like it would be a good example, but honestly they provide so many different formats (JSON, XML, RSS) that I’m not sure you would really want the XML version when RSS is more of a known type and easy to work with.  So let’s look at the Flickr API which is a similarly popular one and has a <a href="http://www.flickr.com/services/api/">well-documented REST interface</a>.  Let’s say you wanted to work with the results of their <a href="http://www.flickr.com/services/api/flickr.interestingness.getList.html">‘interestingness’ public query</a> which will provide you with a list of photos.  We can see in their documentation that <p>Sweet.  Copy that sample response.  Go into Visual Studio in your project class file (or create a new one), go to the edit menu:</p>

<p><img title="Paste XML as Types" alt="Paste XML as Types" src="http://s3.amazonaws.com:80/storage.timheuer.com/pastexmlastypes1.png" /></p>

<p>Booyah!  Watch as the magic happens and the XML structure is transformed into strong types for you.</p>

<p>Well, sorta.  Turns out while I think this is a cool feature, it might have some work still to go.  My first assumption was that the documentation on Flickr matched exactly the response (heck, it says <em>sample response</em>).  But it really is only the response body.  There was some missing response header nodes.  You should call the API directly to see a real response.  Second, even with that it looks like I’m getting some weird namespace stuff.</p>

<p>But regardless of that, even taking an XML file and being able to reflect on that to create an object model <em>on paste</em> is pretty cool.  </p>

<p>Try this out – if you see issues <strong><em><a href="http://aspnet.codeplex.com/Release/ProjectReleases.aspx?ReleaseId=24644">leave comments on the WCF REST Starter Kit site</a></em></strong> so they can see them – you’re welcome to leave them here as well, but I’m not on that team and it’s better to give direct feedback on their project.</p>
