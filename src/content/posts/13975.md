<p>or "find stuff near me" and how i did it. (long read sorry)</p> <p>after <a href="http://timheuer.com/blog/archive/2007/01/31/13974.aspx">my last post</a> about using the proximity search article for sql server 2000, i received some requests for a simple walkthrough on how i implemented my app.  well, here's an attempt at a very simple sample.</p> <p>first, so we are clear, my scenario was as follows: given a known single point, search a database of other known points and return a result set of those points that are known to be within a certain mile radius of the single known point.  better put "find locations near me" :-)</p> <p><br />once you have the locations, there are two key steps here: 1) geocoding those locations and 2) establishing the proximity calculations for the axis points.  the first is rather simple.  you essentially have to use a geocoding service to perform the action.  in my application i use my <a href="http://www.gotdotnet.com/codegallery/codegallery.aspx?id=bcedc8aa-2a05-4eb1-a7ec-24ab21326a12" rel="tag">geocode provider</a> and the yahoo geocode apis.  i've found them to be accurate and current.  google also has some as well.  both are 'free' (certain number of transactions) but you have to register your application.  i think yahoo is also implementing referrer-based checking as well, which might be a consideration.  either way, these will geocode your address.</p>
<blockquote>
<p><strong>suggestion</strong>: you can abstract this from your users very easily.  when they enter an address to save into your application UI, take an intermediary step to verify and geocode the address...once you have the lat/long then you are ready to save it to the database.</p></blockquote>
<p>the next step is calculating the axis.  i'm honestly not going to sit here and pretend to tell you that i'm a geospacial expert on the laws of longitude, because i'm not.  basically the math that is performed makes calculations based on the radius and axis point of the earth.  no, i'm serious.  you'll see later one of the parameters is earth radius.  once you have the known lat/long of your addresses, then you have to calc the axis points.  to simplify this for me, i created three functions in sql server like this one:</p>
<div class="csharpcode"><pre><span class="lnum">   1:  </span><span class="kwrd">CREATE</span> <span class="kwrd">FUNCTION</span> [dbo].[XAxis] </pre><pre><span class="lnum">   2:  </span>    (</pre><pre><span class="lnum">   3:  </span>    @lat <span class="kwrd">float</span>,</pre><pre><span class="lnum">   4:  </span>    @lon <span class="kwrd">float</span></pre><pre><span class="lnum">   5:  </span>    )</pre><pre><span class="lnum">   6:  </span><span class="kwrd">RETURNS</span> <span class="kwrd">float</span></pre><pre><span class="lnum">   7:  </span><span class="kwrd">AS</span></pre><pre><span class="lnum">   8:  </span>    <span class="kwrd">BEGIN</span></pre><pre><span class="lnum">   9:  </span>    <span class="kwrd">RETURN</span> COS(4 * (4 * atn2(1, 5) - atn2(1, 239)) / 180 * @lat) * COS(4 * (4 * atn2(1, 5) - atn2(1, 239)) / 180 * @lon)</pre><pre><span class="lnum">  10:  </span>    END</pre></div>
<p><br />the others are similar and you can get the scripts at the bottom of this post.  once you have those in your database for reusability, you have two options: stored procedure or trigger.  stored procedure (or rather, implementing within your insert/update routine) is probably most ideal.  you want to ensure the axis calculations (and also the geocoding) maintains integrity.  after all, if the user updates the address, you want to update all that information!  so within your stored proc you can simply update the XAxis, YAxis, and ZAxis fields using these functions using something like:</p>
<div class="csharpcode"><pre><span class="lnum">   1:  </span><span class="kwrd">UPDATE</span> Locations <span class="kwrd">SET</span> XAxis = dbo.XAxis(@lat, @long)</pre></div>
<p><br />you could also do this in a trigger (as i was using), but it was pointed out to me (thanks bert) that using an AFTER UPDATE trigger might send this in an infinite loop as my trigger was performing an update statement itself.  stored proc is the way to go if you can.  i've included the INSERT trigger in the files just for your perusal though.</p>
<p>okay, with that done, you should have a sequence of when an address is entered (again possibly adding the intermediary step of geocoding for the user) you now have all the information you need.  now when your user needs to perform a "find near me" query the process is simple.</p>
<p>first, you'll need to geocode their asking point.  again, this is simple and can be accomplished by various geocoding apis.  once you have the lat/long of the asking point, you can feed them to the stored proc that will use your previous proximity information and find the ones nearest you.  here's the stored proc:</p>
<div class="csharpcode"><pre><span class="lnum">   1:  </span><span class="kwrd">CREATE</span> <span class="kwrd">PROCEDURE</span> [dbo].[FindNearby]</pre><pre><span class="lnum">   2:  </span>   @CenterLat <span class="kwrd">float</span>, </pre><pre><span class="lnum">   3:  </span>   @CenterLon <span class="kwrd">float</span>, </pre><pre><span class="lnum">   4:  </span>   @SearchDistance <span class="kwrd">float</span>, </pre><pre><span class="lnum">   5:  </span>   @EarthRadius <span class="kwrd">float</span></pre><pre><span class="lnum">   6:  </span><span class="kwrd">AS</span></pre><pre><span class="lnum">   7:  </span><span class="kwrd">declare</span> @CntXAxis <span class="kwrd">float</span></pre><pre><span class="lnum">   8:  </span><span class="kwrd">declare</span> @CntYAxis <span class="kwrd">float</span></pre><pre><span class="lnum">   9:  </span><span class="kwrd">declare</span> @CntZAxis <span class="kwrd">float</span></pre><pre><span class="lnum">  10:  </span> </pre><pre><span class="lnum">  11:  </span><span class="kwrd">set</span> @CntXAxis = cos(radians(@CenterLat)) * cos(radians(@CenterLon))</pre><pre><span class="lnum">  12:  </span><span class="kwrd">set</span> @CntYAxis = cos(radians(@CenterLat)) * sin(radians(@CenterLon))</pre><pre><span class="lnum">  13:  </span><span class="kwrd">set</span> @CntZAxis = sin(radians(@CenterLat))</pre><pre><span class="lnum">  14:  </span> </pre><pre><span class="lnum">  15:  </span><span class="kwrd">select</span> *,  ProxDistance = @EarthRadius * acos( XAxis*@CntXAxis + </pre><pre><span class="lnum">  16:  </span>  YAxis*@CntYAxis + ZAxis*@CntZAxis)</pre><pre><span class="lnum">  17:  </span><span class="kwrd">from</span>  Locations</pre><pre><span class="lnum">  18:  </span><span class="kwrd">where</span>  (@EarthRadius * acos( XAxis*@CntXAxis + YAxis*@CntYAxis + </pre><pre><span class="lnum">  19:  </span>  ZAxis*@CntZAxis) &lt;= @SearchDistance) <span class="kwrd">AND</span> latitude <span class="kwrd">is</span> <span class="kwrd">not</span> <span class="kwrd">null</span></pre><pre><span class="lnum">  20:  </span><span class="kwrd">order</span> <span class="kwrd">by</span> ProxDistance ASC</pre></div>
<p><br />you'll notice the parameters of radius and earth radius.  since i'm using miles, i use the earth radius of 3961 miles.  you can search on live.com and other places for another number, but this seemed to be a general consensus of the radius of the earth in miles.  i put this in my configuration file in case i needed to change it.  no, not in case the earth changed, but in case it needed to be kilometers.  the SearchDistance param needs to be in the same unit of measurement as the earth radius.  feed those and your lat/long in and you get those near the point -- as well as the approximate distance in the same unit of measure.  boom, you are done.  do with the data as you wish.</p>
<p>using <a href="http://dev.live.com/virtualearth/sdk/">virtual earth</a>, you can easily plot those points as i did in my sample application.  heck using <a href="http://dev.live.com/virtualearth/sdk/" rel="tag">virtual earth</a> you could also geocode for you in your user interface.</p>
<blockquote>
<p><strong>suggestion</strong>: you could present a map in your UI.  have the user enter an address...if found you can have them click on the point to verify and capture that lat/long from virtual earth, then sending back to your procedure and skipping that functional step on the backend.</p></blockquote>
