<p>On the heels of the <strong><a href="http://developer.windowsphone.com/">Windows Phone Developer Tools</a></strong> and <a href="http://silverlight.codeplex.com"><strong>Silverlight for Windows Phone Toolkit</strong></a> releases I saw a lot of exhaling going on in the hallways today.  Apparently Jeff saved his largest one for an avalanche of knowledge on <a href="http://twitter.com/timheuer">Twitter</a> in the late afternoon.  <a href="http://www.jeff.wilcox.name/">Jeff Wilcox</a> is a developer on the <a href="http://www.silverlight.net">Silverlight</a> team and has been working on the Silverlight for Windows Phone initiative as well as the toolkit released today.  He was headed out on vacation but decided to throw out some words of wisdom for Windows Phone developers working in Silverlight.  Here’s some of those nuggets – I wanted to capture them before he left because who knows how long they’ll stay in Twitter.  It reads like a true guide to developing great apps on Windows Phone…</p>  <ul>   <li>Panorama looks nice, but Pivot will offer faster start time.</li>    <li>You can also set a Background image to a Pivot. You won't get the parallax effect, but it is another option.</li>    <li>Be aware of how many pano and pivot items you do have. Memory expands quick when you have a lot of views and images!</li>    <li>Even if you have a 30k compressed JPEG image, at runtime that becomes an uncompressed surface that may take several MBs of memory</li>    <li>Pivot and Panorama can have UI element headers and titles, too, but you'll need to apply your own styling (fonts and sizes)</li>    <li>Beware that UI elements larger than 2000x2000 pixels that are bitmap cached clip on Windows Phone 7. We know it isn't perfect, but beware k?</li>    <li>Setting SelectedIndex before the items are set on a Pivot causes an exception. Wrap in a try/catch or wait for loaded (sorry!)</li>    <li>A slideshow app in 5 minutes: Pivot with null Header and Title and item headers. Beware memory use though.</li>    <li>A lot of people try building 'wizard' screens with panorama &amp; pivot control. Please don't do this! Thx, the "UX gods"</li>    <li>Layout is a killer. But like death, you eventually have to pay it for everything.</li>    <li>So consider delay loading controls and screens. A Panorama with a billion items will take forever to load due to layout.</li>    <li>If you're not using PerformanceProgressBar, I'll send @<a href="http://twitter.com/JustinAngel">JustinAngel</a> after you <a title="http://bit.ly/czqcwJ" href="http://www.jeff.wilcox.name/2010/08/progressbarperftips2/">ProgressBar for Windows Phone 7</a></li>    <li>If your app rocks and starts really quick *on a device* consider not using a splash screen</li>    <li>It's true. Your 6-core machine running the wp7 emulator is NOT indicative of device (single core!) performance. Beware!</li>    <li>We've talked perf before... Content over Resources for images means fast startup time <a href="http://bit.ly/9DhVbd">http://bit.ly/9DhVbd</a></li>    <li>If you're using Panorama, a Resource background will load immediately compared to Content</li>    <li>Remember that for ingestion to the marketplace, your apps need to consume under 90MB of memory</li>    <li>However on devices with &gt; 256MB, its cool to use more *in those cases</li>    <li>long deviceTotalMemory = (long)DeviceExtendedProperties.GetValue("DeviceTotalMemory"); </li>    <li>long applicationCurrentMemoryUsage = (long)DeviceExtendedProperties.GetValue("ApplicationCurrentMemoryUsage");</li>    <li>long applicationPeakMemoryUsage = (long)DeviceExtendedProperties.GetValue("ApplicationPeakMemoryUsage")</li>    <li>Your compositor thread should rock out around 60fps all the time please. <a href="http://twitter.com/search?q=%23wp7dev">#wp7dev</a> perf counters: <a href="http://bit.ly/busJIi">http://bit.ly/busJIi</a></li>    <li>If your UI thread gets pegged, your compositor thread will suffer... remember your BackgroundWorker kids!</li>    <li>Unfortunately the "Analytics" type from desktop Silverlight is not on the phone today, so you can't measure CPU in a regular app</li>    <li>Having a single DispatcherTimer in your app can affect your battery life regardless of interval. Chose wisely</li>    <li>You can set the Foreground property on Pivot to set the title and header text color</li>    <li>Using a map control inside a Panorama or Pivot is not recommended for a number of UX and technical reasons. Navigate to a subpage.</li>    <li>Please use text styles and never hardcode sizes or default fonts. </li>    <li>The panorama/pivot items expect most their contents to have a 12px margin left/right for UX reasons. The default styles have this. </li>    <li>So if you have something in a pano/pivot item with 0 margin &amp; padding, your UX will be funky</li>    <li>Fill rate is super important. Keep it under 2.5 please</li>    <li>What is fill rate? 1.00 means one screen of pixels being rendered every frame.</li>    <li>Check your apps for extra, un-needed background colors on pages, controls, etc.. They impact perf.</li>    <li>That sexy "tilt" effect? Use Peter's behavior <a href="http://bit.ly/90Z1yR">http://bit.ly/90Z1yR</a> and/or check out the MSDN docs</li>    <li>DataTemplates with a bunch of StackPanels and Grids? Try to simplify to a Grid with the right col/rows instead for perf wins. </li>    <li>Unit testing in a quick and dirty way is possible on the Windows Phone thx to the sl unit test fx. <a href="http://bit.ly/a0DWah">http://bit.ly/a0DWah</a></li>    <li>Only use Dispatcher.BeginInvoke when you must. Look at SmartDispatcher (ps old code sry) <a href="http://bit.ly/axHh36">http://bit.ly/axHh36</a></li>    <li>For a "wide" Panorama item, set the item's Orientation to "Horizontal"</li>    <li>Play with the cache and redraw vis. settings to see what's being cached in your app <a href="http://bit.ly/busJIi">http://bit.ly/busJIi</a></li>    <li>Things in a list/scroll viewer are often automatically bitmap cached by Silverlight for Windows Phone runtime</li>    <li>If you have a progress bar with IsIndeterminate="True" in your app, even if its hidden those storyboards are costly! Set to False! </li>    <li>We did work on Windows Phone 7 to move more networking to the background thread - hope it helps</li>    <li>When a Panorama loads, all its items go through a render pass. For pivot, it is done incrementally for neighboring items.</li>    <li>When making web requests, see if the service lets you scope down the fields that are returned for quicker perf (and JSON over XML!) </li>    <li>If you navigate to a subpage, the old page will stick around - so complex pano/pivot pgs stay in memory unless you're proactive</li>    <li>The "app deployment tool" installed with the dev tools lets you run others apps in emulator/device without needing source</li>    <li>We optimize for loading some things from isolated storage. Images from an isostore stream may load faster than a MemoryStream</li>    <li>If your source files have "Black" or "White", you might be doing it wrong. PhoneForegroundBrush, PhoneBackgroundBrush instead! </li>    <li>If your control's dep. property has a change handler, animating that prop. will always happen on the UI thread (no gpu accel.)</li>    <li>Animating Opacity on a CacheMode="BitmapCache" element = compositor thread (GPU!)</li>    <li>In the RTM tools, scroll viewers all have the "bounce" effect automatically</li>    <li>If you ignore the phone's theme (and go all light bg, like the mail app), your scrollbarsmust be retemplated or you won't see them! </li>    <li>Although data binding is not evil, an observable col. with a complex data temp. and 200k items is evil.</li>    <li>The web browser control won't let you NavigateToString until it has loaded.</li>    <li>If you have an app with a lot of different web browser controls, think about consolidating to one, so it only has to load once.</li>    <li>Panorama is designed to be a starting place. Think whitespace. Not tons of data</li>    <li>Free performance win: when you use Panorama the way the UX guidelines recommend, it is faster! <a href="http://bit.ly/9zTxtU">http://bit.ly/9zTxtU</a></li>    <li>Those theme xamls for <a href="http://twitter.com/search?q=%23wp7">#wp7</a>? Yeah they are in %ProgramFiles%\Microsoft SDKs\Windows Phone\v7.0\Design\</li>    <li>Resist the urge to Panorama every app.  It is a sweet UX thing when used right…but not just because.</li>    <li>Resist the urge to iPhone gradient your apps. Think outside the box! Also it avoids color banding...</li> </ul>  <p>So as you can see, he sort of knows what he’s talking about :-).  Go <a href="http://www.jeff.wilcox.name">subscribe to his blog</a> and <a href="http://twitter.com/jeffwilcox">follow him on Twitter</a>.  Other helpful links:</p>  <ul>   <li><a href="http://hanselminutes.com/default.aspx?showID=246">Jeff on Hanselminutes</a> talking about Windows Phone 7 application performance</li>    <li><a href="http://www.microsoft.com/downloads/en/confirmation.aspx?familyId=3a8636bf-185f-449a-a0ce-83502b9ec0ec&amp;displayLang=en">Creating High Performance Silverlight for Windows Phone Applications</a> (whitepaper)</li>    <li><a href="http://channel9.msdn.com/shows/Inside+Windows+Phone/Inside-Windows-Phone-03-Optimizing-Windows-Phone-Silverlight-applications/">Shane Guillet on optimizing Windows Phone Silverlight Applications</a> (video)</li> </ul>  <p>Thanks Jeff for sharing your awesomeness!</p>  <p>Hope this helps!   </p><div style="padding-bottom: 0px; margin: 0px; padding-left: 0px; padding-right: 0px; display: inline; float: none; padding-top: 0px" id="scid:0767317B-992E-4b12-91E0-4F059A8CECA8:cf19eb0f-862f-4a5f-b5f8-542b2e8a5757" class="wlWriterEditableSmartContent"></div>
