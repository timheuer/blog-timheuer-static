<p>I’ve been playing around with the <a href="http://silverlight.net/">Silverlight</a> 3 navigation framework some more (thanks for the comments/thoughts on the <a href="http://timheuer.com/blog/archive/2009/04/03/share-data-between-navigation-pages-in-silverlight-3.aspx">last post about sharing data</a>).  I got a few emails about understanding how the navigation works and people coming up with interesting uses.  Let’s take a moment to explore two of these concepts: out-of-browser navigation and controlling your navigation in your app.</p>  <h2>Navigation Basics</h2>  <p>If you are using Visual Studio 2008 and the Silverlight 3 tools, you’ll notice that when you choose to create a new project (or perhaps you didn’t notice and this will be new to you) that you have an option to create a <em>Silverlight Navigation Application</em> which is a starter template for some basic navigation framework.  This isn’t required to use the navigation framework, but merely a starting point.</p>  <p><img style="display: block; float: none; margin-left: auto; margin-right: auto" title="Silverlight Navigation Application template" alt="Silverlight Navigation Application template" src="http://storage.timheuer.com/navapp1.png" /></p>  <p>Once you have this template and click on the different navigation buttons (and/or add your own as well) you’ll see that the content changes based on the event.  Additionally you should notice that the browser’s back/forward buttons integrate with this functionality and allow you to navigate back/forward using the browser controls as well, honoring the navigation context of the application.</p>  <h2>Out-of-browser Navigation</h2>  <p>So now you’ve finished your app, complete with navigation and also added the functionality to <a href="http://silverlight.net/learn/learnvideo.aspx?video=187318">allow the application to be installed out-of-browser</a>.  Yipee!  Your users install the application offline and boom, they call you and say “how do I go back to where I was?”  Well, ideally you’ve created an application that has good navigation points in it for them, but perhaps you want to add simple forward/back functionality while in out-of-browser mode similar to the in-browser buttons that come with their preferred Internet browsing software.</p>  <p>Guess what?  The navigation framework provides some APIs to help you that are a part of the Frame that you are navigating:</p>  <ul>   <li>CanGoForward/CanGoBack</li>    <li>GoForward/GoBack</li> </ul>  <p>With these two you can emulate the same browser back/forward functionality.  Here’s an example using the default VS2008 navigation template.  In MainPage.xaml I’ve removed the branding information (your.application.name) and added the following (about line 30):</p>  <div style="border-bottom: silver 1px solid; <h2>Controlling your navigation history</h2>

<p>The other comment I’ve got is if people wanted to have ultimate control over their history navigation and have some parts integrate with the browser and other parts not.  Let me introduce you to a property of Frame: <strong>JournalOwnership</strong>.  By default the setting is Automatic, which means that the frame integrates with the browser history journal.  Setting it to something else, like OwnsJournal for example, means that it will not automatically integrate and you are now responsible for that navigation.  </p>

<p>What this means is that you can have a frame that is independent of the browser navigation, but still leverage the power of the framework.  Let’s alter our default template and add one of our navigation pages to include it’s <strong>own</strong> navigation pages, but those we don’t want to be a part of the overall app.  </p>

<p>You’ll see it looks similar (just for demonstration purposes) and includes a Frame.  Notice the JournalOwnership property set to <strong>OwnsJournal</strong>.  Now I also have the same back/fwd links  and functions to control them (GoBack/GoForward) in this sample, but now since I (well, SubFrame) owns the journal, I’m responsible for the navigation.  So here’s what happens:</p>

<ul>
  <li>User navigates to app</li>

  <li>User clicks “about” – browser history appended</li>

  <li>User clicks “sub navigation” – browser history appended</li>

  <li>Within the “sub navigation” page, user clicks “subnav 2” – browser history <strong>not updated</strong></li>

  <li>User clicks “back” within the “subnav 2” page – user is taken back to “subnav 1”</li>
</ul>

<p>So By changing the JournalOwnership, I get the control to manage the journal navigation, but also have to remember that.</p>
