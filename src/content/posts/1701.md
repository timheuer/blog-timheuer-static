<P>a second great tool i've been using lately...llblgen pro (<A href="http://www.llblgen.com/">www.llblgen.com</A>).</P>
<P><EM>first: our team got sick of saying &#8220;LLBLGen Pro&#8221; when referring to the tool and the code it generated...so we affectionately named it &#8220;cool j&#8220; (yeah, you can see the deduction of the name...started out being ll cool j, etc.)</EM></P>
<P>i consider myself always a skeptic.&nbsp; i am used to certain things and usually stick to them.&nbsp; you can imagine my skepticism when a teammate decided we use an ORM tool for data access/code generation.</P>
<P><EM>what!? no way, those things don't work</EM></P>
<P>of course my argument had no weight since i had no evidence to the fact that the ORM tools sucked.&nbsp; my teammate had evaluated a number of them and found LLBLGen Pro to be the best one out there from a feature/support perspective.</P>
<P>as with any ORM tool, your data model has to be solid...so we spent a lot of time on that.&nbsp; once completed, &#8220;cool j&#8220; interpreted everything and generated the required data access layers for us -- and using them is a breeze.&nbsp; our project currently has only one stored procedure due to a long running process.</P>
<P><EM>what!? you aren't using stored procs?! are you an idiot? everyone knows in-line sql is stupid, unsecure, hard to support, etc....</EM></P>
<P>that's what i thought as well (again, my comfort zone) until i really did my research.&nbsp; you see, database is not my strongest point beyond the basics...so i had to see the proof for myself.&nbsp; well, what i found is that all the arguments for stored procs vs. dynamic sql statements aren't entirely valid and are based on a lot of assumptions of how dynamic sql statements are build and executed. (for a good synopsis, check out: <A href="http://www.theserverside.net/news/thread.tss?thread_id=31953">http://www.theserverside.net/news/thread.tss?thread_id=31953</A>)</P>
<P>&#8220;cool j&#8221; is awesome here.&nbsp; you see it leverages sp_executesql so that the statements are cached and it uses parameterized queries.&nbsp; additionally, the dynamic sql generated includes ONLY the updates required, because &#8220;cool j&#8221; has the capability of understanding what attributes of the entities are dirty and need to be updated.</P>
<P>in a nutshell, i'm a believer.&nbsp; this tool (along with others: <A href="http://timheuer.com/blog/archive/2005/02/11/1670.aspx">http://timheuer.com/blog/archive/2005/02/11/1670.aspx</A>) has really helped accellerate my development allowing me to concentrate on core business functionality.&nbsp; yes i'm still comfortable doing certain things my way, but i'm learning more and more about the generated code from &#8220;cool j&#8221; and ways of making it better based on my designs.</P>
<P>check it out.</P>
