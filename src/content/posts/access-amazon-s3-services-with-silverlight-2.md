<p>I ran into an interesting situation last week…the desire to access some of my <strong>Amazon S3 </strong>services from within a <a href="http://silverlight.net"><strong>Silverlight</strong></a> application.</p>
<blockquote>
<p>Amazon Simple Storage Solution (S3) is a pay service provided Amazon for object storage ‘in the cloud.’  Although there is no UI tool provided by Amazon to navigate your account in S3, a SOAP and REST API are available for developers to integrate S3 information into their applications or other uses.  You can view more information about <a href="http://www.amazon.com/S3-AWS-home-page-Money/b/ref=sc_fe_l_2?ie=UTF8&amp;node=16427261&amp;no=3435361&amp;me=A36L942TSJ2AJA">Amazon S3 on their site</a>.</p>
</blockquote>
<p><strong><u>What is S3?</u></strong></p>
<p>Since S3 is a pretty flexible service, it can be used for many different things including storing “objects” like serialized representations of a type or something.  A lot of applications these days are using cloud storage to store their application objects like this.  For me, I use it as a file server in the cloud.  Mostly it is there to host my images on this site as well as downloads and such.  Because there is no user interface I collaborated with Aaron to make the <a href="http://www.codeplex.com/s3browser">S3 Browser for Live Writer</a> so that I could access my files for when I need them in posting content.</p>
<p>Accessing content from S3 also has different meanings, even with regard to Silverlight.  For example, if you wanted to simply set an Image source to a hosted image on S3, you could easily do that using the URL provided by S3 to the object.  Since Silverlight allows media assets to be sourced from anywhere, this is not a problem.</p>
<p><strong><u>The Problem</u></strong></p>
<p>The problem comes when you want to download content in otherways, such as maybe a file stored there, a serialized object, or access their services in a Silverlight application.  Why is this a problem?  Well because S3 does not expose any cross-domain policy files in their implementation.  RIA platforms like Silverlight require a policy file from the providing service to exist in order to make cross-domain calls from the platform.  </p>
<blockquote>
<p>You can read more about Silverlight’s cross-domain policy information here:</p>
<ul>
    <ul>
        <li>VIDEO: <a href="http://silverlight.net/learn/learnvideo.aspx?video=47174">Cross-domain and Silverlight</a> </li>
        <li><a href="http://timheuer.com/blog/archive/2008/04/06/silverlight-cross-domain-policy-file-snippet-intellisense.aspx">Cross-domain policy file helpers</a> </li>

<p> </p>
</blockquote>
<p>So how can we accomplish this?</p>
<p><strong><u>The S3 Service APIs</u></strong></p>
<p>Amazon exposes two service APIs: SOAP and REST.  Because of the requirements around using their REST service and providing an <em>Authorization</em> header, we are unable to use that in Silverlight at this time since <em>Authorization</em> is a restricted header we cannot modify at this time.  So we can use their SOAP service.  This is fine for Silverlight because Amazon provides a <a href="http://s3.amazonaws.com/doc/2006-03-01/AmazonS3.wsdl?">WSDL</a> for us to generate a proxy with.  The defined endpoint in the WSDL for the service is <a href="https://s3.amazonaws.com/soap">https://s3.amazonaws.com/soap</a>.  This is important so remember this.  Let’s move on.</p>
<p><strong><u>Buckets to the rescue!</u></strong></p>
<p>S3 uses a concept they call <em>buckets</em> to store information in containers.  I’m not going to into a lot of detail explaining this concept so if you want to learn more, read their documentation.  Basically a bucket is global unique to the service (not to your account)…so there can only exist one “timheuer” bucket across the service, name them accordingly :-).  All data you push to S3 must be in a bucket.  When you create a bucket, you can also access content in that bucket using a domain shortcut system.  For example when you create a bucket called “timheuer” and put a file in there called foo.txt, it has the URI of http://<strong>timheuer</strong>.<em>s3.amazonaws.com</em>/<strong>foo.txt</strong>.  Notice the alias that is happening here.  We can now use this method to solve some of our issues.  How?  Well the “/soap” key will be available at <strong>any</strong> bucket endpoint!</p>
<p>Because of this aliasing, we can use this mechanism to ‘trick’ the endpoint of our service to respond to the policy file request…so let’s review how we’ll do this.</p>
<p><em>Step 1: Create the bucket</em></p>
<p>There are different ways you could do this.  You could simply put a bucket called “foo” and use that, or you can completely alias a domain name.  I’m choosing to completely alias a domain name.  Here’s how I did it.  First I created a bucket called <em>timheueraws.timheuer.com</em> – yes that full name.  That’s a valid bucket name and you’ll see why I did the full one in a moment.</p>
<p><em>Step 2: Alias the domain</em></p>
<p>If you have control over your <strong><a href="http://en.wikipedia.org/wiki/Domain_Name_System">DNS</a></strong>, this is easy, if you don’t, you may want to use the simple bucket aliasing.  But I went into my DNS (I use <a href="http://www.dnsmadeeasy.com/">dnsmadeeasy.com</a> btw, and it rocks, you should use it too).  I added a CNAME record to my domain (timheuer.com):</p>

<p>What does this mean?  Well any request to timheueraws.timheuer.com will essentially be made at timheueraws.timheuer.com.s3.amazonaws.com.  The last parameter is the TTL (time-to-live).</p>
<p><strong>UPDATE: For security reasons you should actually stick with using only a bucket name and not a CNAME'd bucket.  This will enable you to use the SSL certificate from Amazon and make secure calls.  For example a bucket names "foo" could use <a href="https://foo.s3.amazonaws.com/soap">https://foo.s3.amazonaws.com/soap</a> as the endpoint.  This is highly adviseable.</strong></p>
<p><em>Step 3: Create the clientaccesspolicy.xml file</em></p>
<p>Create the policy file and upload it to your bucket you created in step 1.  Be sure to set the access control list to allow ‘Everyone’ read permissions on it or you’ll have a problem even getting to it.</p>
<p><strong><u>Creating the Silverlight application</u></strong></p>
<p>Create a new Silverlight application using Visual Studio 2008.  You can get all the tools you need by visiting the <a href="http://silverlight.net/GetStarted">Getting Started</a> section of the Silverlight community site.  Once created let’s point out the key aspects of the application.</p>
<p><em>Create the Amazon service reference</em></p>
<p>In your Silverlight application, add a service reference.  The easiest way to do this by right-clicking on the Silverlight project in VS2008 and choosing <em>Add Service Reference</em>.  Then in the address area, specify the Amazon S3 WSDL location:</p>
<p><img alt="" src="http://s3.amazonaws.com:80/timheuer-img/s3-addref.png" /></p>
<p>This will create the necessary proxy class code for us as well as a <em>ServiceReferences.clientconfig </em>file.</p>
<p><em>Write your Amazon code</em></p>
<p>Now for this simple purposes, let’s just list out all the buckets for our account.  There is an API method called “ListAllMyBuckets” that we’ll use.  Now Amazon requires a <em>Signature</em> element with every API call – it is essentially the authentication scheme.  The <em>Signature</em> is a hash of the request plus your Amazon secret key (something you should never share).  This can be confusing to some, so with the perusing of various code libraries on the Amazon doc areas, I came up with a simplified S3Helper to be able to do this Signature generation for us.</p>

<p>So now if we run this application, we should expect to see our bucket list populated in our ListBox, right?  Wrong.  We get a few exceptions.  First, we get an error because we can’t find the cross-domain policy file.  Ah yes, remember we’re still using the Amazon SOAP endpoint.  We need to change that.</p>
<p><strong><u>Changing the Amazon endpoint</u></strong></p>
<p>You may notice that the SOAP endpoint for S3 is an https:// scheme.  We won’t be able to use that using this method because of our aliasing and the fact that the SSL certificate wouldn’t match our alias.  So we need to change our endpoint.  There are two ways we can do this.</p>
<p>We can change this in code and alter our code by creating a new BasicHttpBinding and EndpointAddress and passing them into the constructor of new AWS.AmazonS3Client().  But that would be putting our configuration in code.  Remember that when we added the service reference we were provided with a <em>ServiceReferences.clientconfig</em> file.  Open that up and check it out.  It provides all the configuration information for the endpoint.  Now we could just change a few things.  I decided to create a new &lt;binding&gt; node for my use rather than alter the others.  I called it “CustomAWS” and copied from the existing one that was there.  Now because the default endpoint for S3 is a secure transport and we cannot use that, we have to change the &lt;security&gt; node to mode=”None” so that we can use our custom endpoint URI.</p>
