<p>okay, so it isn&rsquo;t really a fix, but should be.</p>
<p>today the team released the <a href="http://www.asp.net/sandbox/samp_profiles.aspx?tabindex=0&amp;tabid=1">SqlTableProfileProvider</a>&nbsp;which modifies the profile storage mechanism from colon-delimited blobs (which provide for challenging backend reporting) to a method where each property is realized as a separate column.</p>
<p>is this the best implementation? maybe not.&nbsp; a quick look at the docs shows that the table has to be created in advance.&nbsp; one flexibility thing i love about the providers is just that&hellip;the flexibility.&nbsp; in the out-of-box profile provider you can implement new profile properties at will, and not have to touch the database.&nbsp; what this new implementation does is better store the data for reporting, however removes that quick flexibility &mdash; the table would have to be altered each new profile property.</p>
<p>a good start to better storage of that data though!</p>
<p>one gotcha: data isn&rsquo;t serialized and must be compatible with the storage type.</p>
<p>go check out the <a href="http://www.asp.net/sandbox/samp_profiles.aspx?tabindex=0&amp;tabid=1">SqlTableProfileProvider and SqlStoredProcedureProfileProvider</a></p>
<p>also be sure to check out this implementation <a href="http://www.theserverside.net/articles/showarticle.tss?id=CreatingProfileProvider">ASP.NET 2.0 Custom Profile Provider</a> as this is closer (and perhaps great &mdash; i haven&rsquo;t tried it) to what I&rsquo;d be looking for.&nbsp; the data may not be as easy to report on as above, but certainly simpler than colon-delimited.</p>
