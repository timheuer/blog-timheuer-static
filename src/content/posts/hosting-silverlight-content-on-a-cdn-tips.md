<p>Over the holiday this past week I got a ping from <a href="http://weblogs.asp.net/scottcate/">Scott Cate</a> about some <a href="http://silverlight.net">Silverlight</a> media questions, namely a player and <a href="http://silverlight.live.com">Silverlight Streaming</a>.  The gist of the conversation was that the <a href="http://www.microsoft.com/expression/products/Overview.aspx?key=encoder">Encoder 2</a> SP1 templates and the SLS Plugin don’t play nice together (yet).  I know that the team has been testing some updates to the plugin for uploading Silverlight 2 templates to SLS, but for now if you tried to do that you’ll get a random error message that won’t make sense to you (something along the lines of template not found).</p>  <p>After talking with Scott and understanding his scenario more (media formats and CDN hosting – note: <a href="http://en.wikipedia.org/wiki/Content_Delivery_Network"><strong>CDN</strong></a><strong>=Content Delivery Network</strong> - specifically) we, well I, decided it might be a good idea to use an alternate player right now.  Of course I suggested the one that <a href="http://joel.neubeck.net">Joel</a> and I worked on, but there are also many other out there.  I should also point out that the Encoder 2 SP1 templates also include a <strong>fully functional Silverlight 2 player with source code</strong> that you can use.  Scott took my player as a start (not sure if it will be the final one he uses) and we went to chatting about the implementation.  I first told him he could still use SLS and just upload the XAP, etc. but remembered that SLS doesn’t fully support an <em>easy</em> implementation of initParams just yet (it can, but a little workaround-ish right now).  After learning of the initParams, Scott wanted to host the player on his CDN and just pass in the media URI as the param (exactly what our player is built for).</p>  <p>I pointed to the instructions on the <a href="http://sl2videoplayer.codeplex.com">SL2VideoPlayer CodePlex site</a> and we started implementing.  So Scott was going to have his pages delivered from his CMS system on his site (let’s call it foo.com) and have the Silverlight application delivered from his CDN (let’s call that mycdn.com).  Then the media was going to be hosted on another CDN (Silverlight Streaming).  We had three networks in play here: foo.com, mycdn.com and Silverlight Streaming.  In his site delivered by <p>Note that the source for the Silverlight application is a full URI to the CDN endpoint of the XAP file.  Also note the initParams which would point to the video file hosted on Silverlight Streaming (this isn’t the real URI, just a sample).  When rendered, we saw the Silverlight default loader (%age orb) and then a white screen – nothing.  WTF?!  Loading up my <a href="http://timheuer.com/blog/archive/2008/06/10/silverlight-services-cross-domain-404-not-found.aspx">favorite tool for Silverlight network debugging</a> we saw the XAP being delivered and loaded, but then the media file was not.</p>

<p>I then remembered that I may have forgot to tell Scott that any remotely loaded XAP needs to ensure that it is delivered with the correct content-type of <em>application/x-silverlight-app</em> to ensure that it would load properly.  We double-checked this and ensured it was (of course it was because it was already being loaded, but good to double check).  Then we couldn’t figure out why it wasn’t loading the media.</p>

<p>Aha!  I remembered that there was one snippet of code that was in there that does a check for a query string parameter.  In doing this, the Silverlight application uses the HTML DOM bridge to interact with, well, the HTML DOM.  This is what was causing the problem.  <strong>By default, remotely loaded XAP files cannot access the HTML DOM without the author explicitly allowing it</strong>.  So we had to add another parameter to the object tag, the <em>enableHtmlAccess</em> param.  I sent it to Scott in an email (after testing myself) and thought we were done.  He said it still wasn’t working.  Frustrated, we kept looking.  Sergey from the network team alerted me to a possibility of some character encoding issues.  Yep, that was it.  Here’s a screenshot from the code that wasn’t working:</p>

<p><img src="http://storage.timheuer.com/quoteencoding.png" /></p>

<p>Spot the issue?  The copied/pasted param quotes were not in ASCII encoding and thus I’m assuming HTML was ignoring it.  After Scott typed it out rather than pasting, it worked fine.</p>

<p>Phew!  Nothing like banging your head in the weeds of code when it is a little typo!</p>

<p><strong><u>Summary</u></strong></p>

<p>So if you are looking to host your Silverlight content on a CDN for delivery, make sure two things:</p>

<ol>
  <ol>
    <li>The server delivering your XAP content must deliver with the correct MIME type: <em>application/x-silverlight-app</em> </li>

    <li>If there is any code in your XAP that does HTML bridge interaction, you’ll have to add the <em>enableHtmlAccess</em> param to either your object tag instantiation as a param, or to your createObject call if using <a href="http://code.msdn.microsoft.com/silverlightjs">Silverlight.js</a>. </li>
  </ol>
</ol>

<p>I hope this helps some!</p>
