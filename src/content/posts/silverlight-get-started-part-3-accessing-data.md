<p>This is part 3 in a series on getting started with <a href="http://silverlight.net">Silverlight</a>.  To view the <a href="http://timheuer.com/blog/articles/getting-started-with-silverlight-development.aspx">index to the series click here</a>.  <strong>You can download the completed project files for this sample application in </strong><a href="http://storage.timheuer.com/TwitterSearchMonitor_CS.zip"><strong>C#</strong></a><strong> or </strong><a href="http://storage.timheuer.com/TwitterSearchMonitor_VB.zip"><strong>Visual Basic</strong></a><strong>.</strong></p>
<p>Now that we have our initial layout outlined and some controls to work with, let’s start getting the data.  Since we’re going to use <a href="http://twitter.com/timheuer">Twitter</a> search, we’ll be leveraging their web service API.  In our application we won’t be hosting our own database or anything but I did want to point out the various ways you can access data via Silverlight before we go to work on ours.</p>
<h2>Data access options</h2>
<p>One of the bigger beginner misconceptions about accessing data in Silverlight is people looking for some ADO.NET class library in Silverlight.  Stop looking, it isn’t there.  Remember, Silverlight is a <em>client</em> technology that is deployed over the Internet.  You wouldn’t want a browser plug-in to have direct access to your database…as you’d have to expose your database directly to the web.  We all know that is generally a big no-no.</p>
<p>So the next logical step is to expose data via service layers.  This is how Silverlight can communicate with data.  Here are the primary means:</p>
<ul>
    <li>Web services: SOAP, ASP.NET web services (ASMX), WCF services, POX, REST endpoints </li>
    <li>Sockets: network socket communication </li>
    <li>File: accessing static content via web requests. </li>
</ul>
<p><em>Sockets</em></p>
<p>Sockets are probably the most advanced data access endpoints.  These require a socket host to exist and, at the time of this writing, also require communication over a specific port range.  If this is acceptable for you, this can be a very efficient and powerful means of communication for your application.  I don’t think, however, that this is going to be the norm if your application is public facing on the web – I see sockets right now being more for business applications.  Some resources for Sockets:</p>
<ul>
    <li><a href="http://msdn.microsoft.com/en-us/library/cc296248%28VS.95%29.aspx">SDK documentation</a> </li>
    <li><a href="http://weblogs.asp.net/dwahlin/archive/2008/04/13/pushing-data-to-a-silverlight-client-with-sockets-part-ii.aspx">Working with data in sockets</a> (Dan Wahlin, MVP) – this is Silverlight 2 but still relevant to understanding the concepts </li>
</ul>
<p>Working with Sockets requires you to really understand your deployment scenario first before you jump right in and think this will be the best method.</p>
<p><em>File Access</em></p>
<p>Silverlight can communicate with local data or data on the web.  For local data access, the application does not have direct access to the file system, but rather can read/write data via user-initiated actions using the OpenFileDialog and SaveFileDialog APIs to request and save streams of data to the local user’s machine.</p>
<p>Additionally, you can use plain text files or XML files on the web and have your Silverlight application use standard HTTP commands to read/write that information.  Here’s some helper information on using some of these methods:</p>
<ul>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/openfiledialog-and-file-upload">OpenFileDialog and File Upload</a> </li>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/savefiledialog-api">SaveFileDialog</a> </li>
</ul>
<p>You may find yourselves using these techniques to save settings-based data or use very simple data access.</p>
<p><em>Web Services</em></p>
<p>This is the heart of accessing data in Silverlight – through a service layer.  Silverlight supports accessing standard ASP.NET web services (ASMX) or WCF-based services using the familiar <em>Add Service Reference</em> methods in Visual Studio that will generate strongly-typed proxy code for you.</p>
<p>Additionally you can use the standard HTTP verbs to access more POX (Plain old XML) or REST-based endpoints.  Understanding how to consume these various service types is probably the best time spent a developer can educate themselves on understanding what will be right for their scenario.  Here’s some resources:</p>
<ul>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/how-to-consume-wcf-and-aspnet-web-services-in-silverlight">Consume WCF and ASP.NET Services</a> </li>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/calling-web-services-over-https">Calling services over HTTPS</a> </li>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/net-ria-services-intro">.NET RIA Services</a> </li>
    <li><a href="http://wildermuth.com/2009/09/29/Choosing_a_Data_Access_Layer_for_Silverlight_3">Choosing a data access layer</a> (Shawn Wildermuth, MVP) </li>
</ul>
<p>The third point there, .NET RIA Services, is a new framework that aims to make accessing data simpler and more familiar.  The link to the video will walk you through an introduction of that topic.  RIA Services is best when you own the database and are hosting services in the same web application that will be serving up the Silverlight application.</p>

<h2>Cross-domain data access</h2>
<p>Since Silverlight is a web client technology, it operates in the browser’s secure sandbox area and are limited to certain access policies.  One of these restrictions is referred to as cross-domain access.  That is that your application hosted in one domain cannot access services in another domain unless the service says you can.  This “opt-in” approach is commonly known as cross-domain policy files.  Silverlight, like other rich client plug-ins, conforms to these policies.  This is an area that you as a Silverlight developer will likely hit at some point.  Educate yourself sooner than later.  Here are some pointers:</p>
<ul>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/how-to-use-cross-domain-policy-files-with-silverlight">Cross-domain policy files with Silverlight</a> </li>
    <li><a href="http://timheuer.com/blog/archive/2008/04/06/silverlight-cross-domain-policy-file-snippet-intellisense.aspx">Cross-domain policy file helpers</a> </li>
    <li><a href="http://timheuer.com/blog/archive/2008/06/10/silverlight-services-cross-domain-404-not-found.aspx">Troubleshooting cross-domain access with helpful tools</a> </li>
    <li><a href="http://timheuer.com/blog/archive/2008/04/09/silverlight-cannot-access-web-service.aspx">My Silverlight application cannot access my service!</a> </li>
    <li><a href="http://scorbs.com/2008/04/15/silverlight-http-networking-stack-part-2-cross-domain-communication-overview/">Cross-domain communication overview</a> </li>
</ul>
<p>In our Twitter application, we actually will be accessing a service hosted elsewhere and will need to conform to these policies.  Luckily, the <em>search</em> API for Twitter enables this access through their cross-domain policy file (<a href="http://search.twitter.com/crossdomain.xml">http://search.twitter.com/crossdomain.xml</a>). The other areas of Twitter do NOT, which is why, for now, you would not be able to access them directly through Silverlight.  In this situation you would proxy those service calls through your own service that you could enable cross-domain access via a policy file for Silverlight.  Confusing?  It’s simpler than it sounds.</p>
<blockquote>
<p><strong>COMMON MYTH</strong>: You need the Silverlight <strong>and</strong> the Adobe cross-domain policy files in your service to enable access.  This is NOT TRUE and I see it to often people saying <em>I have crossdomain.xml and clientaccesspolicy.xml and it still doesn’t work.</em>  If you are building a service for Silverlight consumption via cross-domain, you only need the clientaccesspolicy.xml file format – that is what we look for first and is the most flexible and secure for Silverlight.</p>
</blockquote>

<h2>Other networking options: HttpWebRequest and ClientHttp</h2>
<p>There are two other network APIs we could have used to access the Twitter API: <a href="http://msdn.microsoft.com/en-us/library/system.net.httpwebrequest%28VS.95%29.aspx">HttpWebRequest</a> and <a href="http://msdn.microsoft.com/en-us/library/system.net.browser.webrequestcreator.clienthttp%28VS.95%29.aspx">ClientHttp</a>.  HttpWebRequest is essentially what we *are* using with WebClient as it is a simple wrapper around that API.  If you needed more granular control over the headers in the request, you’d want to use HttpWebRequest.  Both WebClient and HttpWebRequest make use of the browser’s networking stack.  This presents some limitations, namely not being able to receive all complete status codes or leverage some expanded verbs (PUT/DELETE).  Silverlight has also introduced a ClientHttp option and uses a custom networking stack that enables you to use more verbs as well as receive status code results beyond 200/404.</p>

</ul>

<p>A few things are happening here.  First, the <em>e.Result</em> matches the Stream of response we’ll get from a successfull search.  If an error occurs, we’ll use the ErrorWindow template which is provided for us in the navigation application template we chose.  The <em>_gotLatest</em> member variable helps us track the whether or not we need to reset the max value (which is so future queries request only the latest since the previous query).  After we get the stream we load it into an XmlReader for ease of parsing with our SyndicationFeed class.  SyndicationFeed is a class in the System.ServiceModel.Syndication library that you’ll have to add a reference too in your project.  It has built-in functions for parsing known syndication formats, like RSS and Atom.</p>
<blockquote>
<p><strong>NOTE (Here be dragons):</strong> System.ServiceMode.Syndication brings with it other dependency assemblies.  It is not a small library, but convenient to have.  Take caution in using it for your project and know when and why you need it.  We are using it here so you can be aware of the features and productivity benefits.  An alternative method (especially for just reading syndicated feeds) would be to actually just use LINQ to XML and query the resulting XDocument after loading it.  Again, for demonstration purposes, I wanted to point out the productivity use of SyndicationFeed as a strongly-typed class available to you.  </p>
</blockquote>
<p>More information about reading Syndication data can be found here:</p>
<ul>
    <li><a href="http://silverlight.net/learn/videos/silverlight-videos/working-with-syndicated-data/">Working with Syndicated Data</a> </li>
</ul>
<p>Once we have the SyndicationFeed data loaded, we simply iterate through it and add a new TwitterSearchResult to our ObservableCollection&lt;TwitterSearchResult&gt; object.  You’ll notice we’re doing some conversion on the Image URI to an ImageSource for easier binding later.  Additionally, we’re parsing out the ID of the tweet so we can set the first result (which is the latest) as the most recent ID for later querying (_lastId).</p>
<h2>Giving feedback back to our users</h2>
<p>In our final step here, we want to make sure we are giving some feedback to our users that we are doing something (searching).  Luckily we have something easy for you in an ActivityControl.  At the time of this writing, the ActivityControl was a part of the .NET RIA Services templates, but you can get it <a href="http://www.davidpoll.com/2009/09/14/update-2-displaying-background-activity-in-a-silverlight-ria-application/">here on David Poll’s blog</a>.  You’ll have to build the control and then add a reference to it in your project (if you download the source to our projects the binary is already included in the Libraries folder for you).</p>
<blockquote><p><span style="font-weight: bold;">UPDATE NOTE</span>: While the contents of this tutorial remains unchanged, the ActivityControl is now called the BusyIndicator and is released as a part of the <a href="http://silverlight.codeplex.com">Silverlight Toolkit</a>.  Following the same techniques you can get the official control from the toolkit and have it deployed with your application.  No more need to compile on your own.</p></blockquote>

<p>The flow will be that our Search page will load, start the timer and start the initial search.  After the interval time fires, the search will be executed again, but remember it will use the last known ID to start from as not to load all the repeats again.  This future search data will be added to our ObservableCollection and since the DataGrid is already bound to that, it will be automatically represented in the UI in the appropriate sort order.</p>
<p>We also added some checking to make sure the search term is there and we’re not searching for a blank value.</p>
