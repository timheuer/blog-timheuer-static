<P>when developing <a title="SharePoint" href="http://msdn.microsoft.com/sharepoint" target="_blank">sharepoint</a> web parts and applications, we use the object model and implement the public functions, etc that are available to us...after all there is quite a bit that are available and most are very useful.</P>
<P>what the documentation doesn't always do a good job of, however, is telling us *who* can perform the task.&nbsp; this is a common theme i see when working with customers and one i hit in my first <a title="SharePoint" href="http://msdn.microsoft.com/sharepoint" target="_blank">sharepoint</a> development project *way too late* in the game...we had to completely re-write areas because we made several key assumptions.</P>
<P>developers typically run under administrator privileges in their dev boxes and environments.&nbsp; i'd imagine any <a title="SharePoint" href="http://msdn.microsoft.com/sharepoint" target="_blank">sharepoint</a> dev is an admin or at least a site owner on the <a title="SharePoint" href="http://msdn.microsoft.com/sharepoint" target="_blank">sharepoint</a> site they are working on...maybe this isn't always the best way of doing things in development (see <A href="http://blogs.msdn.com/gduthie/archive/2005/01/19/356543.aspx">andrew duthie's thoughts on least privilege</A>).</P>
<P>similar development caution should be used when writing web parts.&nbsp; <A href="http://blog.u2u.info/DottextWeb/patrick/">patrick </A>points to this <A href="http://barracuda.net/barrysblog.aspx?Date=1/24/2005">article regarding some impersonation techniques </A>to ensure that you can &#8220;get around&#8221; some of these issues.&nbsp; i'd take it one step further and tell you that one thing we did in some instances was to make the <a title="SharePoint" href="http://msdn.microsoft.com/sharepoint" target="_blank">sharepoint</a> application pool account the administrator user (either a box admin or part of the portal admin group).&nbsp; this way rather than maintain our security credentials for a domain user in clear text (in order to use impersonation of a named account in the example you have to), our impersonation code simply &#8220;reverts&#8221; to the process user (which just so happens to be a user with appropriate privileges), then when done...we just revert back.</P>
<P>good stuff.</P>
