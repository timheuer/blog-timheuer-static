also the team bringing you WSL and the new capabilities there!  So you want to try out the terminal…your first question is <em>Where can I get it?</em></p><h2>Building Windows Terminal</h2><p>Well, for now it is source only.  This will likely change over the course of the next few months, but for now if you really want to try it out, you need to build it yourself.  The <a href="https://github.com/microsoft/Terminal/">repository on GitHub</a> has all the bits and instructions for you to do so.  I jumped on this right away like others but ran into a few bumps because I was using Visual Studio 2019.  Luckily this has been solved by <a href="https://github.com/microsoft/Terminal/commit/af7316c130059ba15fe16ff0d13608d956a2b449">this PR</a> to help make the scavenger hunt for dependencies less painful.  </p><blockquote><p>SIDE NOTE: If you have an open source or team-based project, do yourself a favor and add a .vsconfig file now.  Details: <a href="https://devblogs.microsoft.com/setup/configure-visual-studio-across-your-organization-with-vsconfig/?WT.mc_id=blog-blog-timheuer">Configure Visual Studio across your organization with .vsconfig</a></p></blockquote><p>Basically the steps are:</p><ul><li><a href="https://docs.microsoft.com/windows/uwp/get-started/enable-your-device-for-development?WT.mc_id=docs-blog-timheuer">put your machine in developer mode</a></li><li>ensure you have some Git tools</li><li>ensure you have the C++ tools (compilers, etc.) (<a href="https://visualstudio.microsoft.com/?WT.mc_id=visualstudio-blog-timheuer">Visual Studio Community Edition</a> is free to download/install and works)</li><li>clone the repository</li><li><a href="https://github.com/microsoft/Terminal#building-the-code">update the submodules</a></li><li>build the repo (using VS or the tools\bcz script)</li></ul><p>If you use Visual Studio 2019 (recommended by me) opening the OpenConsole.sln file should prompt you with a screen that will give you a note to ‘install’ the missing dependencies in the Solution Explorer:</p><p><img style="margin-right: auto; margin-left: auto; float: none; display: block;" src="https://storage2.timheuer.com/vsconfig-screenshot00.png" /></p><p>If you click Install it will run you through the missing components you may not have enabled for completing the build.  Finish that process.  Once done, you will have all the bits and can go back to the OpenConsole.sln project and choose ‘Build Solution’ and start the process.  Lots of stuff building for the first time so it will take a bit.  Now what?</p><h2>Deploying/Running Windows Terminal</h2><p>Building is step 1, now you want to run it.  Windows Terminal is a Universal Windows Platform (UWP) application and must be deployed first.  The <em>easiest</em> way for doing this is from Visual Studio, but you may want to know what project!?  The <strong>Terminal\CascadiaPackage</strong> project is the one you want and just right-click on that and choose Deploy.  This will deploy it to your machine (assuming you enabled developer mode).  And this will now be in your start menu:</p><p><img style="margin-right: auto; margin-left: auto; float: none; display: block;" src="https://storage2.timheuer.com/startmenuterminal.png" /></p><p>Click/tap/whatever on that and you will launch Terminal for the first time!</p><h2>Wait, where are the tabs I saw?</h2><p>Ah, now that you have it running, let’s take a tour.  The settings right now are a JSON file that you can access via the settings menu.  How do you get the settings menu?  Hit CTRL + T to bring up the tab view and then you should have a drop-down menu in the upper right area:</p><p><img style="margin-right: auto; margin-left: auto; float: none; display: block;" src="https://storage2.timheuer.com/terminalmenupic.png" /></p><p>If you click Settings it will open up the profile.json file in the default editor on your machine configured for editing JSON files.  For me this was Visual Studio, but you can use Code or other editor as well.  This is editing your own profile for your terminal environment.  By default this JSON is void of white-space so you may want to format it to make it more readable.  In Visual Studio that is CTRL + K, CTRL + D and it will prettify it for you.  You’ll then see some initial settings in the top of the file:</p><p /><pre class="brush: css; toolbar: false; highlight: [5];">{

  "experimental_showTabsInTitlebar": false,

what other options exist:</p><ul><li>name: the name of the particular profile.  Right now this is what shows in the drop-down menu shown previously.  There is an issue logged to perhaps make this the now.</p><h2>Navigating the source and contributing</h2><p>As I started this post about how I didn’t like C++, it is a good place to learn a lot.  The Windows Terminal source is a maze of someone else’s code and right now without a clear map.  So the best way is to really just dig in and follow a particular flow.  I find this is the best way to navigate any project source code: find one feature and try to find and follow it in the code.  For me, I was messing with the settings file so much and I hated using the menu with my mouse I wanted a keyboard shortcut.  I knew that CTRL + T launched a new tab, so hey I could implement a keyboard shortcut to launch the settings file quickly.  I first logged that feature as an issue on the project: <a href="https://github.com/microsoft/Terminal/issues/683">Add keybinding for access to Settings</a> and then went to work.  I knew that the new tab shortcut existed so I took my own advice: <em>follow the code!</em>  After a few tries and navigating all the headers and various code files I submitted a PR to the project for my own issue: <a href="https://github.com/microsoft/Terminal/pull/684/commits/c5cadf7b004ab4c380dabfe52c3458935df7383e">PR#684</a>.  There were 6 files changed to add this key binding and if you look at the change I proposed, there was already a function I could call (same one the menu uses) and I really just needed to map the key binding and have it call the existing function.  Simple, but I learned a bit of how the current code is structured in the repository.</p><p>So my recommendation would be to find something that you are seeing as a piece of existing functionality or something you want to add yourself and start looking at searching for where you think it might be.  You’ll quickly find out some of the structure around TerminalSettings, TerminalCore, TerminalControl, etc. and play around with making some changes.  Keep in mind this is active development and things will change around quite frequently.  The team even indicates much:</p><blockquote><p>⚠ <strong>Note</strong>: The Command-Line Team is actively working out of this repository and will be periodically re-structuring the code to make it easier to comprehend, navigate, build, test, and contribute to, so <strong>DO expect significant changes to code layout on a regular basis</strong>.</p></blockquote><p>I have already been bitten a bit by a merge, but no big deal just fix up a few things and move along again!</p><h2>What’s Next?</h2><p>The team has a good aspiration of what they want to accomplish and just getting started.  Quality will improve, bugs will get fixed, consumption of the terminal will be better, etc.  Read
